# v0.6.4 — Tampermonkey sandbox postMessage bridge fix

**Diagnosis from v0.6.3 field signal.** User's console output ruled out 4 of the 5 candidate failure modes in one read — the actual failure was in a different layer entirely.

```
[gh-md-us] userscript loaded v0.6.3 at https://luutuankiet.github.io/gh-md-editor/
[gh-asset] processing https://github.com/user-attachments/assets/…  (×5)
[gh-asset] resolve→userscript https://github.com/user-attachments/assets/…  (×5)
[gh-asset] resolve failed/null → fallback https://github.com/user-attachments/assets/…  (×5, ~15s later)
```

What's missing is what told us everything:

- **No `[gh-asset] heartbeat from userscript v0.6.3`** — userscript-side heartbeat never reached the page handler.
- **No `[gh-md-us] request N <url>`** — page-side resolve request never reached the userscript handler.

Both directions of the postMessage channel were broken. The walker took the resolve path because `window.__ghMdAssetProxy === "0.6.3"` was set on the page (Tampermonkey writes that across the sandbox boundary on script load), so detection passed. But every subsequent message in both directions was being filtered out, and 15 seconds later the page-side timeout fired and produced the 5 stuck fallback shells.

## Root cause

Tampermonkey's default inject mode runs the userscript in a sandboxed JS context. Inside that sandbox, `window` is a JavaScript Proxy wrapping `unsafeWindow` (the page's real window). When the page calls `window.postMessage(...)`, the message lands on the page's real window — but inside the userscript, the MessageEvent's `event.source` references `unsafeWindow`, not the sandbox-proxy `window`. The userscript's listener had:

```js
window.addEventListener('message', (event) => {
  if (event.source !== window) return;  // <-- broken in TM sandbox
  ...
});
```

In sandbox mode, `event.source` (= `unsafeWindow`) !== `window` (= sandbox proxy), so the message was rejected silently. Same failure in reverse: when the userscript called `window.postMessage(...)`, the browser tagged `event.source` as the sandbox proxy, and the page-side handler — also using `event.source !== window` — rejected it.

The `event.source === window` idiom is a common "filter out cross-frame messages" pattern, but it doesn't survive TM's sandbox proxy wrapping.

## Fix

Replace the source-reference check with an origin-string compare on both sides:

```js
// Before (broken in TM sandbox)
if (event.source !== window) return;

// After (immune to window-reference identity)
if (event.origin !== location.origin) return;
```

Same security guarantee (cross-origin messages still rejected), no proxy-vs-real-window comparison. `event.origin` is a string property set by the browser, not affected by sandbox proxies. `location.origin` returns the page's origin even when accessed through the sandbox proxy.

## Userscript side: route through `unsafeWindow` consistently

In addition to the origin-check fix, the userscript now uses `unsafeWindow` for all message I/O — `addEventListener` AND `postMessage`. This pins source-tagging:

- Listener registers on the real page window via `unsafeWindow.addEventListener('message', ...)`. Some TM versions transparently delegate `window.addEventListener` to `unsafeWindow.addEventListener`; some don't. Explicit `unsafeWindow` removes the version-dependent ambiguity.
- Outgoing postMessages via `unsafeWindow.postMessage(...)` tag `event.source` as the real page window, so the page-side handler's filter passes regardless of TM mode.

```js
const W = (typeof unsafeWindow !== 'undefined') ? unsafeWindow : window;
W.addEventListener('message', handler);
W.postMessage(data, '*');
```

Fallback to `window` if `unsafeWindow` isn't exposed (strict Greasemonkey 4+).

## Diagnostic visibility

The page-side heartbeat receive log now fires on EVERY heartbeat arrival, not just the first detection — so the user can confirm the channel is healthy without DevTools archaeology. The userscript also logs every outgoing heartbeat send (`[gh-md-us] heartbeat sent`) and every PING received (`[gh-md-us] ping received → reply heartbeat`) so the channel is observable from both ends.

## What changes from the user's POV

After installing v0.6.4 and hard-refreshing:

- Status pill goes from gray → green within ~1s of page load if the resolve path works end-to-end.
- Console shows `[gh-md-us] heartbeat sent`, `[gh-asset] heartbeat from userscript v0.6.4`, `[gh-md-us] request N <url>`, `[gh-md-us] response 200 … size=… mime=image/png … for <url>`, `[gh-asset] resolved ✓ <url>` — the full happy-path trail.
- If GM.xmlHttpRequest itself fails (e.g. session expired → 401), the failure surfaces via the existing v0.6.3 logs (`[gh-md-us] non-200 401 … finalUrl=…`) and the pill goes red with the specific failure reason. From that signal v0.6.5 (if needed) ships a real fix for the GM-level failure mode.

## What this release is NOT

This release does not change the resolve protocol, the GM.xmlHttpRequest call, the `@connect` scope, or the URL regex. The only change is message-channel plumbing.

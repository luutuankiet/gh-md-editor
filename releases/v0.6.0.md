# v0.6.0 — render private GitHub user-attachments via Tampermonkey companion

GitHub's private/org-scoped image attachments — the kind you get when you paste a screenshot into a PR body on a private repo — require a session cookie to fetch. A static SPA can't send cookies cross-origin from `luutuankiet.github.io` to `github.com`. So previously, `<img src="https://github.com/user-attachments/assets/...">` tags in your markdown rendered as broken-image icons.

v0.6.0 ships a Tampermonkey **companion userscript** that bridges this gap. Install it once; from then on, any private GitHub user-attachment in your markdown renders inline in the preview pane.

## How it works

```
Markdown:
  <img src="https://github.com/user-attachments/assets/abc123">

Preview pane post-render walker (page-side):
  Detects user-attachments URL → swaps src to inline spinner →
  postMessage to userscript → awaits response

Tampermonkey userscript (on YOUR machine, has cookie access):
  GM.xmlHttpRequest GET to github.com WITH cookies
  → GitHub returns 302 → short-lived S3 presigned URL
  → userscript follows redirect, downloads bytes (responseType: 'blob')
  → creates blob: URL → postMessage back to page

Preview pane:
  Receives blob URL → swaps img src → image renders inline
```

No backend, no proxy server, no PATs. Your session cookie stays on your machine; the userscript runs in your browser; image bytes never leave the browser.

## Install

The installer page lives at `/gh-md-editor/userscript-installer.html` — it auto-detects whether Tampermonkey is present, then offers a one-click install link. Tampermonkey intercepts the `.user.js` URL and shows its install dialog.

## Graceful fallback

Without the userscript installed, user-attachments tags render as a `View attachment on GitHub →` link styled to match the editor chrome. The link opens in a new tab where GitHub's own session can authorize the view. No broken images, no surprises.

## Empirical validation

Architecture was validated end-to-end before implementation:

- 5 real org-asset URLs downloaded successfully via `curl -L` with a Firefox session cookie (sizes 54–250 KB, all PNG bytes verified — sample: 1812×670 RGBA).
- PAT-based approaches do NOT work — GitHub returns 404 for `Authorization: Bearer <PAT>` on the user-attachments endpoint regardless of token format (Bearer / token / ?access_token= / Accept: image/*). This endpoint is session-cookie-only by design.
- CDN is `github-production-user-asset-6210df.s3.amazonaws.com` with 5-minute JWT (`X-Amz-Expires=300`) and `Cache-Control: max-age=2592000` (30 days). Browser caches aggressively after first resolve.
- S3 omits `Access-Control-Allow-Origin`, so `fetch()` cross-origin from the page WOULD fail CORS — but `<img src>` rendering doesn't enforce CORS, and the userscript bypasses CORS entirely via `GM.xmlHttpRequest`.

## Privacy

The userscript only touches URLs matching `^https://github\.com/user-attachments/(assets|files)/`. It cannot exfiltrate cookies (Tampermonkey only attaches them on outbound requests the userscript itself initiates). Source lives in `public/userscript.user.js` — open it before installing.

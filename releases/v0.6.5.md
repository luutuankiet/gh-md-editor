# v0.6.5 — Image aspect-ratio fix for explicit width/height attributes

**The bug.** v0.6.4 successfully resolved private GitHub user-attachments images via the Tampermonkey bridge. Console showed the full happy-path trail (heartbeat both ways, 5 requests, 5 responses 200 with sizes 322 KB – 2.0 MB, 5 resolved blob URLs); pill went green. But the rendered images were squashed — stretched horizontally relative to their natural aspect ratio.

**Diagnosis from raw HTML.** GitHub emits user-attachment imgs with explicit `width` AND `height` HTML attributes:

```html
<img width="1024" height="953" alt="image"
     src="blob:..."
     data-gh-asset-processed="true"
     class="gh-asset-resolved">
```

The `github-markdown-css` package the editor uses for preview styling applies `max-width: 100%` to imgs in `.markdown-body`, which clamps them to the pane width. But it does NOT set `height: auto`. So when the browser clamps the width via max-width, the height stays at the fixed `height="953"` attribute value — producing an absurd aspect ratio (e.g. `600px wide × 953px tall` instead of `600px × 558px`).

GitHub's own production CSS includes `.markdown-body img { height: auto; }` precisely to override the height attribute when CSS clamps the width. The npm `github-markdown-css` package omits this rule.

## Fix

One CSS line added to `Preview.svelte`:

```css
:global(.markdown-body img) {
  height: auto;
}
```

Now the browser:
1. Sees `width="1024"` attribute as a presentational hint.
2. Sees `max-width: 100%` from `github-markdown-css` and clamps the rendered width to the container.
3. Sees `height: auto` from our new rule and computes the height from the image's natural aspect ratio rather than the static `height="953"` attribute.

Result: 600px wide × 558px tall, correct aspect.

## Why this didn't surface earlier

The bug pre-existed v0.6.0. Any user with inline `<img width="X" height="Y">` in their markdown would have hit it. But until v0.6.4 made private user-attachments actually resolve to viewable bytes, the canonical test case (an org PR markdown with raw `<img>` tags emitted by GitHub) didn't produce a visible image — it produced fallback links. v0.6.4 unlocked the rendering, which exposed the aspect-ratio bug behind it.

## Files changed

- `src/components/Preview.svelte` — added `:global(.markdown-body img) { height: auto; }` to the scoped style block. Comment explains the github-markdown-css gap.
- `package.json`, `public/userscript-installer.html` — version + footer bump 0.6.4 → 0.6.5.

## What this release is NOT

The userscript is unchanged — no `@version` bump. Users on the v0.6.4 userscript don't need to update Tampermonkey; the fix is page-side CSS only. Hard-refresh the editor and the new CSS lands.

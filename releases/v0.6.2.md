# v0.6.2 — late-heartbeat fallback recovery

**Bug:** if the page-side walker ran BEFORE the userscript heartbeat was received (a timing race that depends on bundle parse speed, network, and Tampermonkey's injection order), the walker fell back to the no-userscript path, wrapped each `<img>` in a `<span class="gh-asset-shell-fallback">`, and locked the image into the "View attachment on GitHub" link state. The userscript heartbeat would arrive seconds later — but the walker only re-evaluates imgs on morphdom diffs, which don't fire without a markdown edit. So fresh page loads with images at the top stayed stuck in fallback even though the userscript was correctly installed and active.

**Fix:** in `src/lib/gh-asset-resolver.ts`, the heartbeat handler now detects the heartbeat-arrived-after-first-render case. On the FIRST heartbeat (when `isUserscriptInstalled()` would have returned false at the time of the message), it calls a new `reprocessFallbackImages()` which:

1. Finds all `.gh-asset-shell-fallback` elements currently in the DOM
2. Moves the wrapped `<img>` back to the shell's parent at the shell's position
3. Removes the shell
4. Clears `data-gh-asset-processed` on the img
5. Hands the img to `processOne()`, which now takes the spinner → resolve → blob URL path

Re-entry guard prevents recursion if heartbeats fire while reprocess is running.

Idempotent on subsequent heartbeats — once there are no fallback shells, the function is a no-op.

**Other:** userscript `@version` and project version both bumped to 0.6.2 for UX consistency. Installer page footer updated.

**Result:** even if the heartbeat arrives 5 seconds after the walker fired fallback, the images self-rescue. No tab reload required.

# v0.7.2 — mermaid multi-line labels no longer bleed below rectangles

v0.7.1 fixed the cramped/uniform-shrink text on wide mermaid diagrams. But it surfaced a latent layout bug for nodes with 3+ line `\n`-escaped labels: the bottom line(s) visually bleed below the node rectangle border. Reproduced exactly on the user's bug-report flowchart (CLI Input → Orchestrator → Interviewer Pipeline …) where every 3-line node showed text cut off below the box.

## Root cause (from user-pasted live DOM dump)

Mermaid v10 with our existing config sizes each label's `<foreignObject>` correctly at ~19 px per line using its own font metric. The user dumped the actual rendered SVG which proved it: `foreignObject width="272.5" height="57.5"` for a 3-line label (57.5 ÷ 3 = 19.17 px per line) — mermaid's measurement is fine.

The bleed happens because `github-markdown-css` cascades `line-height: 1.5` into `.markdown-body` text, which inherits into the foreignObject's inner `<div>` and `<span class="nodeLabel">`. At default ~16 px font + 1.5 line-height, the browser renders each line at ~24 px. Three lines = 72 px of actual text in a 57.5 px foreignObject → ~14 px of overflow per multi-line label. The mermaid <rect> is sized to match the foreignObject (correct from mermaid's POV), so the overflow appears OUTSIDE the rect border.

## Fix

Three-line CSS rule scoped to `.mermaid-block` that resets `line-height: 1.2` on the foreignObject inner div / span / .nodeLabel selectors. Browser then renders each text line at ~19 px (matching mermaid's measurement), labels fit inside their foreignObjects, no bleed.

No mermaid config change. No JS change. The v0.7.1 useMaxWidth per-diagram-type setting stays in place — it correctly produces natural-width SVGs that overflow horizontally for wide diagrams.

## Highlights

| Area | Change |
|---|---|
| `src/app.css` | NEW rule: `.mermaid-block foreignObject > div, .mermaid-block foreignObject > div > .nodeLabel, .mermaid-block foreignObject > div > span, .mermaid-block .edgeLabel, .mermaid-block .nodeLabel { line-height: 1.2; }` |

## Why this took three failed attempts to find

The initial diagnostic loop tried (1) reverting `useMaxWidth: false`, (2) normalizing `\\n` source escapes to real newlines, (3) switching `securityLevel: 'strict' → 'loose'`. None changed the output — because none touched the actual bug, which lives in CSS inheritance, not mermaid config or layout. Without browser JS-eval to read the actual rendered SVG, I couldn't see that mermaid was sizing foreignObjects correctly — the rendered text was just laid out taller by the browser due to CSS leakage. The user pasted the live SVG dump and the bug became obvious in seconds.

Lesson: when mermaid rendering looks wrong inside a styled markdown body, ALWAYS suspect inherited CSS (line-height, font-size, font-family, letter-spacing) on the foreignObject's inner content before blaming mermaid's layout. Mermaid uses its own font metric to size; the browser uses cascaded CSS to render; any divergence between the two produces visible misalignment.

## Verify

`npm run build` exit 0. Visual: same wide flowchart pasted into editor renders with EVERY multi-line node properly containing all its `\n`-split lines within the rectangle border. No bleed below. Single-line and 2-line nodes unaffected (their natural rendered height was within the foreignObject already).

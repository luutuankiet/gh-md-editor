# v0.5.2 — Alt+Click multi-cursor: explicit handler for cross-browser parity

**Released:** 2026-05-21
**Type:** hotfix (one-line behavior change, no API surface)
**Predecessor:** [v0.5.1](./v0.5.1.md)

---

## The one thing this fixes

`Alt+Left-click` now drops a new cursor at the pointer position, on **every** browser — Chromium, Firefox, WebKit — across **every** OS — macOS, Linux, Windows.

In v0.5.1 we flipped `EditorState.allowMultipleSelections.of(true)` and assumed CodeMirror's built-in pointer handler would take care of the rest. `Cmd+D` (next-match) worked. `Alt+Click` did not, specifically on **macOS + Firefox**. The flag is necessary but not sufficient.

## Root cause

CodeMirror 6 wires Alt+click multi-cursor through its internal `mouseSelectionStyle` facet — the same facet that handles word-double-click and drag-to-select. On macOS, **Firefox intercepts `Option+click` at the browser layer** for its own purposes:

- Smart word-by-word drag-select (a Firefox-specific affordance)
- Accessibility shortcuts that vary by user system prefs

By the time the event reaches CM6's listener on `cm-content`, either `altKey` is false (translated away) or the click has already been pre-empted. The net effect: CM6's add-cursor branch never runs, even though the multi-selection state is enabled and ready to receive new ranges.

## The fix

Intercept `mousedown` at the DOM level via `EditorView.domEventHandlers` — the same hook we already use for `contextmenu` (right-click reveal) and `keydown` (Mac Opt+Z wrap toggle). When the gesture is unambiguously “plain Alt+Left-click”, take over completely:

```ts
mousedown: (event, vw) => {
  if (event.button !== 0) return false;
  if (!event.altKey || event.shiftKey || event.ctrlKey || event.metaKey) return false;
  const pos = vw.posAtCoords({ x: event.clientX, y: event.clientY });
  if (pos == null) return false;
  event.preventDefault();
  const existing = vw.state.selection;
  const newCursor = EditorSelection.cursor(pos);
  vw.dispatch({
    selection: EditorSelection.create(
      [...existing.ranges, newCursor],
      existing.ranges.length,
    ),
  });
  return true;
}
```

Key behaviors:

- **Modifier policy is strict.** ONLY plain `Alt+Left-click` matches. Any of `Shift` / `Ctrl` / `Meta` also held → we pass through and let CM6 handle it (those are reserved for native chords: `Shift+click` = extend, `Cmd+click` on macOS = native, etc).
- **Right-click is unaffected.** `event.button !== 0` early-exits, so the existing `contextmenu` handler (right-click → reveal preview) keeps working.
- **The new cursor becomes primary.** `mainIndex` is set to `existing.ranges.length` (the index of the newly-appended range), matching VS Code: typing happens at the most-recently-clicked cursor.
- **Preserves existing selection ranges.** If you had a selection or multiple cursors already, they all survive; the new cursor is added on top.

## What's still the same

- `Cmd/Ctrl+D` (next-match cursor) — unchanged, still works via the v0.5.1 path (`selectNextOccurrence` from `searchKeymap`).
- All other shortcuts — unchanged.
- Sample doc, shortcuts dialog — unchanged.

## What we considered and rejected

- **Adding `drawSelection({drawRangeCursor: true})`** — that flag affects rendering, not event handling. Doesn't fix the Firefox/Mac interception.
- **Patching CM6 internals via a custom `mouseSelectionStyle` facet** — too brittle, ties us to CM6 internals that aren't part of the documented stable API.
- **`event.stopImmediatePropagation()` to also block CM6's native handler** — not needed. Returning `true` from a `domEventHandlers` handler signals “consumed” and CM6 doesn't run subsequent handlers for that event.

## Edge cases worth knowing (and where the limits are)

- **Alt+click inside a code fence**: works — the markdown syntax tree doesn't gate the gesture, only the modifier check does.
- **Alt+click on the line-number gutter**: the gutter is outside the content area, `posAtCoords` returns `null`, handler returns false, browser default runs. No regression.
- **Alt+click drag (Alt held, then drag to select rectangular)**: still bound to CM6's default. Our handler fires on `mousedown` only — the subsequent drag motion still flows through CM6's selection-style machinery. If Mac Firefox also intercepts drag-with-Alt, the rectangular-select gesture may not work on that browser; rectangular-select is not promoted as a feature here, so this is acceptable.
- **Double-Alt+click**: each click adds a cursor at that exact spot. Two cursors stacked. Consistent with VS Code (it doesn't deduplicate either).
- **Existing cursor at the exact pointer position**: we still append, leaving two ranges at the same offset. CM6 collapses these into one on the next selection-change; no visible artifact. (Toggle-to-remove is a v0.6 polish if we ever want VS Code parity on that exact edge.)

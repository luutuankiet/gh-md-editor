# v0.6.3 — Diagnostic instrumentation + visible userscript status pill

**The bug we're chasing.** A private GitHub user-attachments image stuck on the `[View attachment on GitHub →]` fallback link even though the Tampermonkey companion userscript is freshly installed and confirmed running. Verification from DevTools shows the right things in the right places:

```
> window.__ghMdAssetProxy
"0.6.2"
> document.querySelectorAll('.gh-asset-shell-fallback').length
5
```

That combination rules out the timing race v0.6.2 fixed: the walker DID detect the userscript on first paint, took the resolve path, and ended up in the `else { applyFallback() }` branch. The bytes never came back from `GM.xmlHttpRequest`. Something is failing during the round trip and we have **zero observability** into which step — the resolver and the userscript are silent.

**What this release does.** It does NOT fix the resolve failure — we don't yet know what's failing. It puts logs at every transition and a visible status pill on the editor so the next round of triage takes seconds instead of a DevTools dive.

## Userscript side (`@version` 0.6.2 → 0.6.3)

Every `GM.xmlHttpRequest` state now logs to the page console with the `[gh-md-us]` prefix:

```
[gh-md-us] userscript loaded v0.6.3 at https://...
[gh-md-us] request 1 https://github.com/user-attachments/assets/...
[gh-md-us] response 200 412ms size=153022 mime=image/png finalUrl=https://... for https://github.com/user-attachments/assets/...
[gh-md-us] non-200 401 Unauthorized 230ms finalUrl=https://... for https://github.com/user-attachments/assets/...
[gh-md-us] network-error 51ms status=0 finalUrl=... for https://...
[gh-md-us] timeout 15000ms for https://...
```

That `finalUrl` field is the load-bearing piece: it tells you whether the redirect to `*.amazonaws.com` happened and where the actual failure landed.

## Page side (`gh-asset-resolver.ts`)

The page-side bridge logs heartbeat receive, every walker invocation, every resolve attempt, and every outcome:

```
[gh-asset] heartbeat from userscript v0.6.3
[gh-asset] processing https://github.com/user-attachments/assets/...
[gh-asset] resolve→userscript https://github.com/user-attachments/assets/...
[gh-asset] resolved ✓ https://github.com/user-attachments/assets/...
[gh-asset] resolve failed/null → fallback https://github.com/user-attachments/assets/...
[gh-asset] heartbeat-late: reprocessing 5 fallback shells
```

## Visible status pill — bottom-right of the editor

A new `UserscriptStatus.svelte` component pinned to the bottom-right corner, subscribed to a new `resolverStats` Svelte store exported from `gh-asset-resolver.ts`:

| State | Color | Label |
|---|---|---|
| no userscript heartbeat seen | gray | `userscript not detected` |
| resolving N attachments | yellow (pulsing) | `resolving N…` |
| all resolved | green | `userscript v0.6.3 · N resolved` |
| some failed | red | `userscript v0.6.3 · N ok · N failed` |

Click the pill to expand a failure list with the exact reason (`http-401`, `network-error`, `timeout`, `resolve-null-or-timeout`) and the URL that broke. No DevTools required for the first level of triage.

## What to expect after upgrading

1. Hard-refresh the editor.
2. Accept the Tampermonkey update prompt to pick up v0.6.3 of the userscript (versions must match — only the v0.6.3 userscript emits the `[gh-md-us]` logs).
3. Hard-refresh again so the v0.6.3 userscript runs on the next page load.
4. Open a PR with private user-attachments images.
5. Check the status pill in the corner. If red, click it for the failure breakdown.
6. Open DevTools → Console; filter for `gh-md-us` or `gh-asset` to see the full trail.

## What this release is NOT

This release intentionally ships no resolve-failure fix. The plan is to use the new instrumentation to pin down which of the candidate failure modes is firing in the field:

- 401/403 → cookies aren't reaching `github.com` (TM `@connect` scope or session expired)
- Redirect to `*.amazonaws.com` → CDN signed URL response (success or fail, but at least the redirect happened)
- `network-error` with `finalUrl` containing `amazonaws` → CDN refused the request
- `timeout` → request hung; possibly slow network or TM stuck
- `non-asset-url` rejection → walker handed a URL the userscript regex doesn't recognize

v0.6.4 will be a surgical fix once the field data narrows the suspect list.

## Files changed

- `src/lib/gh-asset-resolver.ts` — added `resolverStats` writable store + `ResolverStats` / `ResolverFailure` types; instrumented heartbeat handler, `processOne`, and `reprocessFallbackImages` with console logs and store updates. Stale failure entries are cleared when a heartbeat-late retry runs.
- `public/userscript.user.js` — bumped `@version` and `VERSION` to 0.6.3; added `[gh-md-us]`-prefixed logs at load, request, response (200), non-200, network-error, timeout, and blob-create failure. Logs include duration (`ms`) and `finalUrl` for redirect debugging.
- `src/components/UserscriptStatus.svelte` — new component. Subscribes to `resolverStats`; renders the status pill bottom-right of the editor; expands on click to show counters and the last 10 failure rows. Keyboard accessible (Enter/Space to expand). Dark-mode styled.
- `src/components/App.svelte` — imported and slotted `<UserscriptStatus />` next to `<ShortcutsDialog>` (both render outside `<main>` as overlays).
- `package.json`, `public/userscript-installer.html` — version bumps to 0.6.3 for consistency.
